第一个 Sprint 的目标是实现 **MVP（最小可行产品）**，即打通“用户注册 -> 登录 -> 选车 -> 模拟支付 -> 确认预订”的完整闭环 。

------

## 1. 用户管理：账户与登录 (ID 1)

这是系统的入口，根据要求，用户必须注册才能使用系统 。

- **注册功能**：系统需提供注册页面，收集必要的用户信息如用户名、密码（5-10位）、电子邮件（不需要验证） 并存入数据库 。
- **登录验证**：用户通过凭据登录，系统需建立会话（Session）或令牌（Token）机制来识别已登录用户 。
- **权限控制**：登录后，系统应能区分“客户界面”和“管理界面”，确保普通用户无法访问配置功能 。
- **响应式界面**：注册和登录页面需适配移动端，并符合基本的可访问性标准（如颜色和字体选择） 。

## 2. 预订流程 (IDs 4, 5, 6, 8)

这是应用的核心业务逻辑，必须在 Sprint 1 中通过自动化测试验证其稳定性 。

### A. 查看租赁选项与成本 (ID 4)

- **价格展示**：系统必须清晰显示四种租赁时长的成本：**1小时、4小时、1天、1周** 。
- **实时获取**：价格数据应从后台数据库动态获取，而非前端硬编码，以便管理者随时修改 。

### B. 预订 e-scooter (ID 5)

- **车辆选择**：用户必须能选择特定的电动滑板车 ID 。
- **时长选择**：用户从 ID 4 定义的四个固定时段中选择一个租赁期限 。
- **状态检查**：系统需验证所选滑板车在请求的时段内是否可用（虽 ID 10 是低优先级，但在 MVP 中应具备基础的占用逻辑判断） 。

### C. 模拟支付处理 (ID 6)

- **支付触发**：在用户确认预订详情后，触发模拟支付流程 。
- **模拟逻辑**：无需对接真实银行网关，但需模拟输入卡片信息的过程，并根据规则返回“支付成功”或“支付失败”的状态 。
- **数据记录**：支付成功后，系统需记录交易金额、日期和关联的用户 ID 。

### D. 存储与显示预订确认 (ID 8)

- **持久化存储**：预订成功后的数据（预订 ID、车辆 ID、时段、费用）必须存入数据库 。
- **按需显示**：用户应能在其账户页面（客户界面）随时查看当前的预订状态和历史记录 。

## 3. 基础管理功能：配置滑板车与成本 (ID 16)

这是管理界面的核心，由系统管理员（Manager）操作 。

### 详细功能描述：

- **车辆信息管理**：
  - **添加车辆**：管理员可以向系统中添加新的电动滑板车，设置唯一的车辆 ID 和初始位置信息 。
  - **编辑/删除**：支持修改现有滑板车的详情或从系统中移除车辆 。
- **定价配置**：
  - **费率设置**：管理员可以动态设置和修改 1hr, 4hrs, 1day, 1week 这四个租赁时段的收费标准 。
  - **即时生效**：修改后的价格应立即反映在客户的租赁选项页面中（对应 ID 4） 。
- **界面要求**：
  - 由于是管理后台，应优先保证功能的准确性和数据的可维护性，同时符合响应式 Web 设计要求 。
